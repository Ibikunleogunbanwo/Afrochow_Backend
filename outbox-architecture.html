Afrochow — Transactional Outbox Architecture

Event-driven notification pipeline with crash-safe delivery

⚡ Before Spring Events

DB commit → publishEvent() → Async thread
Async thread → NotificationService → Email / In-App
⚠ If the app crashes between the DB commit and the async thread firing, the event is gone forever. No retry. No record. Customer never gets their order confirmation.

🛡 After Outbox Pattern

DB commit → outbox_event row (same tx)
OutboxPoller → reads PENDING → dispatches
✓ Crash after commit? The outbox row survives. On restart the poller finds it and notifications go out. At-least-once delivery guaranteed by the DB.

🔒Atomicity
The outbox row and the business state change (order/payment/review) are written in the same DB transaction. Either both commit or both roll back. No phantom notifications.
🔁Retry Logic
Failed dispatches (email service down, DB timeout) are retried up to 3 times. After that the row moves to FAILED status and stays in the DB for manual inspection or replay.
⚖️Concurrency Safe
SELECT FOR UPDATE SKIP LOCKED means if you ever run two app instances behind a load balancer, they'll never process the same outbox row simultaneously.
🚀RabbitMQ Path
If you grow to multiple services, swap OutboxPoller to publish messages to Rabbit instead of calling NotificationService directly. Zero changes to publishers.
